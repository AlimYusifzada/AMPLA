## File Format Compatibility

The ampla library is built around a key pattern: Parsing text-based industrial configuration files into a graph of interconnected Python dictionary nodes.

File Input (.AAX/.BAX) ──> Parsed into ──> Class: AAX / BAX
                                                          │
                                                Maintains a dictionary of
                                                          ▼
                                                  Class: block
                                                  (Address/Pins/Value)


The program is designed to compare pairs of **.AAX**, **.AA**, **.BA**, or **.BAX** files. Using other formats may result in runtime errors.

**Possible Combinations:**

* AAX <-> AAX
* AAX <-> AA
* AA <-> AA
* BAX <-> BAX
* BA <-> BAX
* BA <-> BA

> Code files can be from different sources, i.e., one can be extracted from FCB and the other from ONB.

> The program automatically detects BA/BAX/AA/AAX files.

> If the database or code is split into several files, they will be combined together.

Comparison is performed block by block, not line by line.

The program attempts to read and compare numeric constants as numbers. For example, `D=2.3e-3` and `D=0.0023` are considered the same and will not be detected as a difference.

If the XLS report file name ends with `*_DIF*`, it indicates that discrepancies have been found. XLS reports are generated by `GenXLSreport(code_before, code_after)`.

## 🛠️ Detailed Class Breakdown
1. The Primitive Unit: class block

Every logical function (like AND, OR, MOVE) or database record (like an analog input connection) is represented as an instance of the block class.

    Key Attributes:

        self.Address: The primary unique hardware key (e.g., "PC10.3.2.1").

        self.Name: The type or opcode of the element (e.g., "MOVE").

        self.Pins: A Python dictionary ({ pin_name: pin_value }) mapping inputs/outputs to variables or other addresses.

        self.LineNumber: Tracking reference pointing to the exact line in the source file.

    Special Magic Methods Implemented:

        __str__(self): Formats the block and all of its pins into a clean, human-readable terminal output.

        __eq__(self, other): Evaluates deep equality by comparing names, pins, descriptions, and structural routing.

        __add__(self, values): Operator overload allowing quick pin population: my_block += (":1", "PC10.1.1.1:2").

    Core Method:

        Compare(self, other): Performs an element-by-element, pin-by-pin audit between this block and a modified variant, returning clear structural differences.

2. The Project Aggregators: class AAX & class BAX

These classes read, combine, and house the entire parsed infrastructure of a controller.

    AAX (Execution Logic Parser):

        Processes execution pathways, track networks, and execution order loops.

        Organizes items directly by their execution coordinates into the self.Blocks dictionary.

    BAX (Database Instance Parser):

        Inherits directly from AAX.

        Overrides the Parse() ruleset to extract tag databases, raw hardware instance bindings, and register declarations.

    Key Methods Available:

        Read(): Automatically stitches multi-part sequential log splits (e.g., reading App01.AAX, then following through to App02.AAX seamlessly if no terminating END statement is detected).

        Parse(): Scans strings to dynamically populate metadata parameters and clean complex multi-line array-style pin values.

        GetBlock(blkey): Safely returns a target block instance from memory or returns a NONE constant wrapper with an explicit warning.

        xRef(tag): Global search mechanism. Returns a tuple of every address where a variable tag or system name is wired or targeted.

        GetLabels(): Sweeps structural text registers extracting custom tag annotations (e.g., variables beginning with N=).

        Compare(other): Executes an automated controller baseline-to-baseline diff report tracking matching keys, mismatched configurations, or dead/dangling elements.

3. Binary Layout Extensions: class AA & class BA

Because some ABB code revisions deploy compressed or packaged structures instead of flat Unicode files, ampla provides AA and BA extensions. They leverage a low-level binary interpreter routine (readA) to decompress hex arrays directly into standard text layouts before running the layout engine parsers.

## Requirements for Using Source Code

For users who are using the source code instead of the compiled version, please ensure the following:

* Use **Python 3**.
* `ampla.py` is the core file containing all functions for parsing and code comparison.
* External package requirements: **tkinter**, **xlwt**, **datetime**.

To compile the python code to exe use pyinstaller

```bash
pip install pyinstaller
pyinstaller amplscope.py -d all --onefile -i aaxcmp.ico -w --splash amplscope.png --hidden-import=pyi_splash
```

## Python Example to Compare .AA and .AAX Files

```python
from ampla import *

aa_before = AA('source_code_before.aa')

aa_after = AAX('source_code_after.aax')

print(aa_before.compare(aa_after))

GenXLSreport('source_code_before.aa','source_code_after.aax')
```


## Project Structure Overview
The Proj class manages multiple source codes from a single controller.
Object Hierarchy nodeX (Proj instance) acts as a dictionary of PC programs.
Example: nodeX = Proj(path_to_SRCE)

pc10 (Program instance) Accessed via nodeX["PC10"].
Acts as a dictionary of logic blocks within that specific PC program.

BLOCK (Logic Block object) Accessed via pc10["PC10.3.2.1"]. Contains the specific block logic and data.

```python
# 1. Initialize the project controller
nodeX = Proj(path_to_SRCE)

# 2. Access a specific PC program
pc10 = nodeX["PC10"]

# 3. Retrieve the target logic block object
logic_block = pc10["PC10.3.2.1"]
```
